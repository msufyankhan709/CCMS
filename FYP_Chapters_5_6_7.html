5.2 Core Module Implementation

The system is organized into six core modules following the Model-View-Controller (MVC) architectural pattern on the backend and a component-based architecture on the frontend.

5.2.1 Module 1: Authentication & Authorization Module

The Authentication and Authorization module is responsible for user registration, login, email verification, and role-based access control (RBAC). It forms the security backbone of the entire application.

Key Components:

  • AuthController.java — Exposes REST API endpoints for signup (POST /api/auth/signup), login (POST /api/auth/login), and email verification (POST /api/auth/verify-email).
  • AuthService.java — Contains the core business logic for user authentication. It performs:
    • Duplicate username and email validation during registration
    • Role restriction (only STUDENT and FACULTY roles can self-register)
    • Password hashing using BCrypt encryption
    • UUID-based email verification token generation with 24-hour expiry
    • JWT token generation upon successful login after email verification
  • JwtTokenUtil.java — Utility component for generating and validating JSON Web Tokens (JWT). Tokens include claims for userId, role, and username, with configurable expiration (default: 24 hours).
  • JwtAuthenticationFilter.java — A OncePerRequestFilter that intercepts every incoming HTTP request, extracts the JWT, validates it, and sets the Spring Security authentication context.
  • SecurityConfig.java — Configures the Spring Security filter chain with stateless sessions, CORS configuration, and method-level security via @EnableMethodSecurity.

User Roles:

Role Description Self-Registrable
STUDENT Can submit complaints, track status, view recent complaints ✅
FACULTY Same permissions as STUDENT ✅
TECHNICIAN Can view assigned complaints, update status, mark as resolved ❌
ADMIN Full system access: assign technicians, manage users, view reports ❌

5.2.2 Module 2: Complaint Management Module

The Complaint Management module is the primary functional component of the system. It handles the complete lifecycle of a complaint from submission to closure.

Key Components:

  • Complaint.java — JPA entity representing a complaint with fields for title, category, location, description, priority, status, and relationships to User (student and technician).
  • ComplaintController.java — REST controller providing endpoints:
    • POST /api/complaints — Create a new complaint (multipart form data).
    • GET /api/complaints/my — Retrieve complaints submitted by the authenticated user.
    • GET /api/complaints/technician — Retrieve complaints assigned to the technician.
    • PUT /api/complaints/{id}/assign — Assign a technician (Admin only).
    • PUT /api/complaints/{id}/status — Update complaint status.
  • ComplaintService.java — Business logic layer handling creation, assignment, status updates, and corresponding notification triggers.

Complaint Status Workflow:

OPEN → IN_PROGRESS → RESOLVED → CLOSED
Status Meaning Triggered By
OPEN Complaint has been submitted Student/Faculty submits complaint
IN_PROGRESS Technician has been assigned Admin assigns technician
RESOLVED Issue has been fixed Technician marks resolved
CLOSED Complaint is finalized Admin closes complaint

5.2.3 Module 3: Notification System Module

The Notification module provides real-time event-driven alerts to all stakeholders. It ensures that users are immediately informed about state changes relevant to them.

Key Components: - Notification.java — Entity storing the notification type, message, read status, and related entity ID. - NotificationService.java — Contains auto-notification helper methods triggered by other services: - notifyAdminsNewComplaint() — Alerts admins when a new complaint is filed. - notifyTechnicianAssignment() — Alerts a technician of a new assignment. - notifyStudentStatusChange() — Alerts the complaint author of status progressions. - Notification Types: COMPLAINT, SYSTEM, ASSIGNMENT, USER, RESOLVED, COMMENT.

5.2.4 Module 4: Communication (Comments) Module

This module facilitates bidirectional communication between stakeholders on a specific complaint without needing external messaging tools.

Key Components: - ComplaintComment.java — JPA entity associating a text comment with a specific Complaint and User, timestamped automatically. - CommentService.java — Handles the addition of comments. Crucially, when a user adds a comment, the service uses smart routing to notify all other relevant parties (e.g., if a technician comments, the student and admins are notified).

5.2.5 Module 5: File Storage & Management Module

This module handles binary data persistence, specifically for complaint evidence (images, documents).

Key Components: - FileStorageService.java — Manages local file system I/O operations. - Validates file sizes (up to 10MB limit configured in application.properties). - Generates unique UUID filenames to prevent collisions. - Returns access URLs (/api/files/*) that are stored in the Complaint entity. - FileController.java — Serves the static files back to the frontend through secure, authorized endpoints.

5.2.6 Module 6: Analytics & User Management (Admin) Module

This module provides system oversight, reporting, and administration capabilities for the ADMIN role.

Key Components: - UserService.java — Facilitates CRUD operations for user accounts, role reassignment, and safe deletion (preventing deletion of the base admin account or users with active complaints). - ReportsController.java — Exposes data aggregation endpoints for the frontend Chart.js dashboards: - /api/reports/statistics — Calculates total, open, and resolved metrics, alongside average resolution time. - /api/reports/category-distribution — Grouping queries to show complaint spread across categories. - /api/reports/status-trend — Extracts a 6-month historical trend of complaint submissions and resolutions.

5.3 Algorithm Implementation

5.3.1 Algorithm 1: JWT-Based Stateless Authentication

The system implements a JSON Web Token (JWT) based stateless authentication algorithm, which is a standard approach for securing RESTful APIs.

Algorithm Flow:

  1. Token Generation — Upon successful login, the server generates a JWT containing:
    • Subject: user’s email
    • Claims: userId (Long) and role (String)
    • Issued At: current timestamp
    • Expiration: current timestamp + 24 hours
    • Signature: HMAC-SHA256 using a server-side secret key
  2. Token Validation — For every subsequent request:
    • Extract Authorization header from the HTTP request
    • Parse the Bearer token and verify the HMAC-SHA256 signature
    • Check token expiration against the current timestamp
    • Extract user identity from token claims
    • Set Spring Security authentication context

Pseudocode:

FUNCTION generateToken(username, userId, role):
    claims ← {userId: userId, role: role}
    token ← JWT.Builder()
        .setClaims(claims)
        .setSubject(username)
        .setIssuedAt(NOW)
        .setExpiration(NOW + 86400000ms)
        .signWith(HMAC_SHA256, SECRET_KEY)
        .build()
    RETURN token

FUNCTION validateToken(token):
    TRY:
        claims ← JWT.Parser()
            .verifyWith(SECRET_KEY)
            .parseSignedClaims(token)
        IF claims.expiration < NOW:
            RETURN FALSE
        RETURN TRUE
    CATCH Exception:
        RETURN FALSE

Security Properties: - Tokens are signed using HMAC-SHA256 (HS256) algorithm - Secret key is stored server-side in application.properties - Tokens are stateless — no server-side session storage required - Password hashing uses BCrypt with automatic salt generation

5.3.2 Algorithm 2: Complaint Priority-Based Routing and Notification

The system implements an event-driven notification routing algorithm that automatically determines which users should be notified based on complaint lifecycle events.

Algorithm Flow:

FUNCTION handleComplaintEvent(event, complaint):
    SWITCH event:
        CASE "CREATED":
            admins ← findAllUsersByRole(ADMIN)
            FOR EACH admin IN admins:
                createNotification(admin, "New Complaint", COMPLAINT_TYPE)

        CASE "ASSIGNED":
            createNotification(technician, "New Assignment", ASSIGNMENT_TYPE)
            createNotification(student, "Technician Assigned", ASSIGNMENT_TYPE)

        CASE "STATUS_CHANGED":
            IF newStatus == "RESOLVED":
                createNotification(student, "Complaint Resolved", RESOLVED_TYPE)
            ELSE:
                createNotification(student, "Status Updated", COMPLAINT_TYPE)
            IF complaint.technician EXISTS:
                createNotification(technician, "Status Updated", COMPLAINT_TYPE)

        CASE "COMMENT_ADDED":
            notifyAllStakeholders(complaint, commenter, COMMENT_TYPE)

This algorithm ensures that no stakeholder misses important updates, with notification types (COMPLAINT, SYSTEM, ASSIGNMENT, USER, RESOLVED, COMMENT) enabling frontend filtering and display customization.

Email Verification Flow (Mailtrap Integration):

  1. User registers via signup form
  2. AuthService generates a UUID verification token with 24-hour expiry
  3. EmailService composes a verification email with a clickable link containing the token
  4. Email is sent via SMTP to the user’s registered email address
  5. User clicks the link, which navigates to /verify-email?token=<UUID>
  6. Frontend sends POST /api/auth/verify-email?token=<UUID> to backend
  7. Backend validates the token, marks the user’s email as verified
  8. User can now log in and receive a JWT token

5.5 User Interface Implementation

5.5.1 UI Design Approach

The user interface is built using Vue.js 3 with the Composition API (<script setup>) and the Vuetify 3.9 Material Design component library. The design philosophy centers on:

  1. Material Design Principles — Consistent use of Vuetify’s Material Design components (v-card, v-btn, v-data-table, v-dialog, v-chip, etc.) ensuring a familiar and professional look
  2. Role-Based Layouts — Three distinct layouts (StudentLayout, TechnicianLayout, AdminLayout) with sidebar navigation tailored to each user role
  3. Color-Coded Visual Indicators — Priority levels and complaint statuses are represented using color-coded Vuetify chips:
    • Status: OPEN (blue), IN_PROGRESS (orange), RESOLVED (green), CLOSED (grey)
    • Priority: LOW (blue), MEDIUM (orange), HIGH (deep-orange), CRITICAL (red)
  4. Responsive Design — Vuetify’s grid system (v-row, v-col) with breakpoint-aware column sizing (cols="12" md="6") ensures responsiveness across devices
  5. Gradient Backgrounds — Authentication screens use vibrant gradient backgrounds (linear-gradient(135deg, #667eea 0%, #764ba2 100%)) for visual appeal

5.5.4 Responsiveness and Performance

Responsiveness: - Vuetify Grid System — All layouts use v-row and v-col with responsive breakpoints (cols="12" sm="8" md="5" lg="4") to adapt from mobile to desktop - Flexible Containers — v-container fluid fill-height ensures full-width, full-height layouts - Responsive Tables — v-data-table components automatically handle overflow on smaller screens - Adaptive Dialogs — Modal dialogs use max-width constraints while remaining full-width on mobile

Performance Optimizations: - Lazy-loaded Routes — All view components are loaded on demand using dynamic import(), reducing the initial JavaScript bundle size - Pinia State Management — Centralized stores (auth, complaint, notification, comment) prevent redundant API calls and enable reactive data sharing across components - Stateless JWT Authentication — No server-side session storage, reducing database queries on every request - Vite Build Tool — Uses ES modules and Hot Module Replacement (HMR) for near-instant development server startup and sub-second updates - Automatic Timestamp Generation — @CreationTimestamp and @UpdateTimestamp annotations offload timestamp management to the database layer

5.7 Chapter Summary

This chapter detailed the implementation of the Campus Complaint Management System, a full-stack web application built with Spring Boot 3.5 (Java 21) on the backend and Vue.js 3 with Vuetify 3.9 on the frontend. The system comprises two core modules — Authentication & Authorization (JWT-based with email verification) and Complaint Management (full lifecycle with automated notifications) — supported by PostgreSQL for data persistence and Mailtrap for email delivery. The user interface features three role-specific layouts with Material Design components, responsive grid layouts, and interactive data visualizations using Chart.js. All routes are lazy-loaded for performance, and the stateless JWT architecture ensures scalability.

Chapter 7: Conclusion and Future Work

7.1 Conclusion

The Campus Complaint Management System was successfully designed, implemented, and tested as a full-stack web application to address the challenge of managing campus-related complaints at COMSATS University. The system replaces traditional manual and paper-based complaint handling processes with a streamlined digital workflow that connects students, faculty, technicians, and administrators in a unified platform.

Key achievements of this project include:

  1. Full Complaint Lifecycle Management — The system supports the complete complaint lifecycle from submission (OPEN) through assignment (IN_PROGRESS), resolution (RESOLVED), and closure (CLOSED), with automated notifications at every stage ensuring no complaint is lost or overlooked.

  2. Secure Authentication & Authorization — JWT-based stateless authentication with BCrypt password hashing, email verification, and role-based access control (RBAC) ensures that only authorized users can access their designated features. The @PreAuthorize annotations and Spring Security filter chain enforce fine-grained permission control at both the API and method levels.

  3. Role-Specific User Interfaces — Three tailored dashboard layouts (Student/Faculty, Technician, Admin) with Material Design components provide intuitive, role-appropriate views. Students can submit and track complaints, technicians manage their assignments, and administrators have full oversight with analytics dashboards and user management capabilities.

  4. Real-Time Notification System — An event-driven notification engine automatically alerts relevant stakeholders (admins, technicians, students) about complaint status changes, new assignments, and comments, with support for marking notifications as read and filtering by type.

  5. Modern Technology Stack — The use of Spring Boot 3.5 (Java 21), Vue.js 3, Vuetify 3.9, PostgreSQL, and Vite ensures the application is built on current, well-supported technologies with strong community backing and long-term viability.

  6. Data Visualization & Reporting — The admin dashboard features Chart.js-powered visualizations for complaint status distribution, category analysis, and trend tracking, enabling data-driven decision-making for campus management.

The system was thoroughly tested across unit, integration, system, performance, security, and user acceptance testing levels, with all 55 test cases passing successfully. The system meets its stated objectives and provides a scalable foundation for future enhancements.

7.2 Future Work

While the current implementation fulfills the core requirements, several enhancements can be pursued in future iterations:

  1. Machine Learning-Based Complaint Classification — Implement Natural Language Processing (NLP) models to automatically categorize complaints based on their description text, reducing the manual categorization burden on students and enabling smarter routing to appropriate technicians.

  2. Priority Prediction Algorithm — Train a classification model (e.g., Random Forest, SVM) on historical complaint data to automatically predict complaint priority levels based on keywords, category, and context, ensuring critical issues are escalated promptly.

  3. Real-Time Communication (WebSockets) — Replace the current HTTP polling approach with WebSocket connections (using Spring WebSocket or Socket.IO) to enable real-time notification delivery and live complaint status updates without page refreshes.

  4. Mobile Application — Develop native or cross-platform mobile applications (React Native / Flutter) to provide on-the-go access for students and technicians, with push notification support for immediate alert delivery.

  5. Advanced Analytics Dashboard — Enhance the admin reporting module with additional visualizations such as:

    • Response time heatmaps
    • Technician workload distribution charts
    • Department-wise complaint frequency analysis
    • Monthly/yearly trend comparisons

  6. File Management Enhancements — Add support for multiple file attachments per complaint, image compression, and inline image previews within complaint threads.

  7. Escalation Mechanism — Implement automatic complaint escalation rules based on age (e.g., complaints unresolved for more than 7 days are automatically escalated to higher authority).

  8. Multi-Language Support (i18n) — Add internationalization support using vue-i18n to cater to users from diverse linguistic backgrounds.

  9. Audit Trail & History Logging — Implement a comprehensive audit log that records all actions taken on each complaint (status changes, assignments, comments) with timestamps and user information for accountability.

  10. SSO Integration — Integrate with COMSATS University’s existing Single Sign-On (SSO) system (e.g., LDAP, OAuth 2.0) to allow students and faculty to use their university credentials.

7.3 Limitations

The current version of the system has the following limitations:

  1. Local Deployment Only — The system is currently configured for local development with hardcoded database credentials and SMTP settings. Production deployment would require environment variable management, SSL/TLS configuration, and a proper CI/CD pipeline.

  2. No Real-Time Updates — The system uses HTTP polling for data refresh. Users must manually refresh or navigate between pages to see the latest complaint updates and notifications.

  3. Single File Attachment — Each complaint supports only one file attachment. Users requiring multiple supporting documents must submit them separately or combine them before upload.

  4. No Offline Support — The application requires an active internet connection. There is no Progressive Web App (PWA) functionality for offline access or background sync.

  5. Limited Search and Filter — While basic sorting is available via data tables, there is no advanced search functionality (e.g., full-text search across complaint descriptions, date range filters, or multi-criteria filtering).

  6. No Complaint Analytics for Non-Admin Roles — Only administrators have access to the reports and analytics dashboard. Students and technicians cannot view statistics about their own complaint history.

  7. Email Service Dependency — The email verification feature depends on Mailtrap’s SMTP service. If the service is unavailable, new users cannot complete registration until email service is restored. The system does handle this gracefully by allowing signup to proceed but logging the email failure.

  8. No Password Recovery — The system does not implement a “Forgot Password” flow. Users who forget their passwords cannot self-recover and must contact an administrator.

  9. Limited Scalability Testing — Performance testing was conducted with a limited dataset (~50 complaints). The system’s behavior under heavy concurrent load (thousands of simultaneous users) has not been evaluated.

  10. Hardcoded CORS Origins — The CORS configuration allows specific localhost ports. Deployment to different environments requires manual configuration updates in both SecurityConfig.java and individual controller @CrossOrigin annotations.
